### 명령어	상세 설명
아래 명령어들을 브랜치를 만들며 사용해 보았습니다.  
1) `git remote -v`	현재 로컬 저장소가 연결된 **원격 저장소(Remote)**의 이름과 주소(URL)를 보여줍니다. Codespaces에서는 자동으로 origin으로 설정됩니다.
2) `git status	현재` 작업 디렉토리와 스테이징 영역의 상태를 보여줍니다. 어떤 파일이 수정되었는지, 커밋할 준비가 되었는지 등을 확인합니다.
3) `git branch`	로컬 저장소에 존재하는 모든 브랜치 목록을 보여주며, * 표시로 현재 작업 중인 브랜치를 나타냅니다.
4) `git checkout -b [이름]`	새 브랜치를 생성하고, 생성과 동시에 해당 브랜치로 작업 환경을 전환합니다.
5) `git checkout [이름]`지정된 브랜치로 작업 환경을 전환합니다.

스테이징이란?  
Git의 3단계 구조(작업 디렉토리 -> **스테이징 영역** -> 로컬 저장소) 중 중간 단계입니다.  
`git add` 명령어를 사용하여 작업한 파일 중 **특정 변경 사항만** 다음 커밋(스냅샷)에 포함되도록 준비하는 곳입니다. 커밋하기 전에 어떤 내용을 저장할지 미리 확정하는 임시 공간입니다.

```bash
# 4. 변경 사항을 스테이징 영역에 추가
# 커밋할 변경 사항을 확정합니다. 여기서는 README.md만 스테이징합니다.
git add README.md

# 이후 git status를 입력하면 상태가 바뀐 것을 확인할 수 있습니다.  
```

### 4.  두 번째 브랜치 작업 및 커밋/푸시

```bash
git checkout feature/config-settings
```

<img width="918" height="352" alt="image" src="https://github.com/user-attachments/assets/c896f04a-419a-479b-8d56-b75cd670ac78" />  

아까 다른 브랜치에서 작업했던 내용이죠? 근데 왜 작업한 내용이 없을까요?   
Git에서 **브랜치**는 독립된 작업 공간을 의미합니다. `git checkout`으로 브랜치를 전환하면, Git은 해당 브랜치가 마지막으로 커밋된 시점의 파일 상태로 작업 디렉토리를 완전히 되돌립니다.  
따라서 `feature/user-details`에서 작업하고 커밋한 내용은 **해당 브랜치에만** 저장되어 있고, 현재 `feature/config-settings` 브랜치는 그 내용을 알지 못하기 때문에 변경 사항이 보이지 않는 것입니다.  
  
```bash
# 3. 변경 사항을 스테이징 및 커밋 (git add . 명령어로 모든 변경 사항 한 번에 스테이징)
git add .
git commit -m "feat: 설정 구성 섹션 초안 추가"
```
아까는 git add ??.md로 ~ 파일만 저장(커밋)했습니다. 하지만 . 를 쓰면 지금 작업 공간에 있는 모든 파일들을 저장할 수 있습니다.
리눅스 명령어에서 .는 현재 경로를 나타내죠? 마찬가지로 git add . 를 입력하면 현재 경로에 있는 파일들을 다 스테이징할 수 있습니다.  
<img width="874" height="337" alt="image" src="https://github.com/user-attachments/assets/422e5053-c78f-4889-b619-bd18d73777f3" />  
 

지금까지 feature/config-settings에서 작업한 내용을 GitHub에서 볼 수 있나요? 아직 볼 수 없습니다.     
아래 단계인 push를 하면 볼 수 있게 됩니다.  

```bash
# 4. 원격 저장소에 푸시
# 로컬 브랜치(feature/config-settings)를 원격 저장소에 올립니다.
git push origin feature/config-settings
```
